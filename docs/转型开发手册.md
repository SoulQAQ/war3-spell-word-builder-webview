# 转型开发手册

本手册用于指导后续 Agent 在**保留当前工程技术方案与交付方式**的前提下，将业务从“WC3 技能文本生成”转型为其他领域应用。

---

## 1. 目标与边界

### 1.1 转型目标
- 保留技术栈、工程结构、UI 视觉语言、打包与启动方式。
- 仅替换业务模型、业务文案、业务流程与数据字段。
- 保证桌面端可运行、可保存、可导入导出、可打包。

### 1.2 不应改变（默认禁止）
- 启动方式：`start-uv.bat` / `start.bat`
- 主入口：`script/gui.py` + `webui/index.html`
- 桌面框架：`pywebview`
- 后端语言与版本约束：Python 3.13
- 打包流程：`build.bat` 与 PyInstaller 路径处理机制

---

## 2. 必须保留的工程基线

### 2.1 技术栈基线
- 前端：Vanilla HTML/CSS/JS（单页、无框架）
- 后端：Python + pywebview
- 数据：本地 JSON 文件
- 配置：`config/setting.yaml` + `data/settings.json`

### 2.2 目录与职责基线
- `script/gui.py`：后端 API 与桌面入口
- `webui/index.html`：前端 UI、状态、交互、API 调用
- `data/`：业务数据、配色方案、导出文件
- `docs/`：架构与设计文档

### 2.3 UI 风格基线
- 保留当前卡片式布局、侧栏树、底部操作栏、Toast 通知。
- 保留 CSS 变量体系（色彩、边框、阴影、圆角、间距）。
- 保留左侧预览/控制区与右侧编辑区的整体结构。

---

## 3. 可替换内容（转型核心）

以下内容允许按新业务重做：
- 业务实体：如“技能”可替换为“模板/条目/规则/任务”等
- 字段定义：ID、名称、属性、描述、参数等
- 预览逻辑：字符串拼接、渲染模板、规则解释
- 导入导出格式：JSON 结构可调整
- 树节点语义：分类/条目命名可调整

---

## 4. API 设计约定（强制）

### 4.1 命名规范
- 继续使用 `wc3_*` 前缀或统一迁移到新前缀（二选一）。
- 若迁移前缀，必须保留兼容别名，避免前后端命名不一致。

### 4.2 返回格式
- 所有 API 必须返回统一结构：
```json
{ "success": true|false, "data": {}, "message": "" }
```

### 4.3 兼容策略
- `webui/index.html` 的 `API_METHOD_ALIASES` 必须与 `script/gui.py` 同步。
- 若 API 不可用，只允许在“开发浏览器模式”做非阻塞 fallback。
- 桌面真实运行环境下，API 必须真实可调用，不可仅隐藏报错。

---

## 5. 转型实施步骤（推荐顺序）

1. 定义新业务数据模型（JSON Schema + 默认值）
2. 修改后端 API：读写/树管理/导入导出/设置
3. 保留 UI 框架，仅替换字段与文案
4. 更新预览与生成逻辑
5. 更新导入导出兼容层（含旧数据迁移）
6. 校验启动、保存、另存为、导入、导出、打包

---

## 6. 回归高风险点（必须重点检查）

历史上频繁回归的问题，转型时必须回归测试：
- 初始化 API 名称不一致导致 `API method not found`
- 保存 API 名称不一致导致无法保存
- 左侧预览控制区出现被 `flex` 拉伸的大面积空白
- 表单首行输入框溢出导致横向滚动条
- 右键菜单功能接入后事件冒泡导致误触发

---

## 7. 数据迁移与兼容要求

- 老数据加载失败时，必须有字段默认值兜底。
- 缺失字段用迁移函数补齐，不允许让 UI 直接崩溃。
- 导入失败要给明确错误信息，不可静默失败。
- 如调整参数语法，需保留旧格式解析兼容至少一个版本周期。

---

## 8. 打包与部署规范

### 8.1 本地运行
- 推荐：`start-uv.bat`
- 备用：`start.bat`

### 8.2 打包
- 使用既有 `build.bat`/PyInstaller 方案
- 保证 `webui`、`config` 等资源路径在 `frozen` 和开发环境都可用
- 禁止改坏 `sys.frozen` / `_MEIPASS` 路径逻辑

---

## 9. Agent 执行规范（建议直接复用）

后续 Agent 接到“业务转型”任务时，默认执行：
1. 先阅读：`README.md`、`docs/PROJECT_ARCHITECTURE.md`、`docs/TECHNICAL_DESIGN.md`
2. 先做 API 对照表：前端调用名 vs 后端注册名
3. 先做最小改造：保留布局与交互壳，只替换业务字段
4. 改完必须做启动与核心流程验证
5. 输出必须包含：
   - 修改文件列表
   - 保留项说明
   - 新业务模型说明
   - 回归测试结果

---

## 10. 交付验收清单

- 能启动（`start-uv.bat`）
- 初始化成功，无 API not found
- 树可增删改查，右键菜单可用
- 当前条目可保存/另存为
- 导入/导出成功
- 页面无非预期横向滚动
- 打包脚本可执行

---

## 11. 版本记录建议

每次“业务转型”建议新增一条变更记录：
- 转型目标
- 影响范围（API/UI/数据/打包）
- 兼容策略
- 回归结果

建议记录位置：`docs/转型开发手册.md` 末尾追加或单独建 `docs/migration-log.md`。
